# Design: #489 — DJ request-list → client-side sort/filter (both event-page lists)

**Issue:** #489 (follow-up to #478)
**Date:** 2026-06-20
**Author:** thewrz (design steered via brainstorming)
**Scope:** Frontend only (`dashboard/`). No backend, SSE, or migration changes.

## Why

#478 added **server-side** pagination + sorting for the DJ event request list
(paginated envelope, true `total`, growing-window "Load More"). For a **single
event's** request data — a bounded set (tens to low thousands of rows) — the
better architecture is **client-side**: load the full set into the browser once,
then sort / filter / count entirely in memory, and refresh on live updates.

Wins (all delivered by "load full set once, sort/filter/count in JS"):

- **Instant UX** — sort/filter is a sub-millisecond `Array.sort`; no per-toggle
  round-trip (today a sort change waits for a server fetch, `page.tsx:492`).
- **Counts correct by construction** — tab counts/totals come from the in-memory
  set, so they can never be window-limited (structurally fixes the #411/#478
  "counts reflect only what's loaded" failure mode).
- **Lower per-interaction server load** — no re-query on every sort/filter change.

## Consistency scope (the real point of this issue)

The `events/[code]` page hosts **two** bounded request lists, **both currently on
the #478 server-side sort/offset model**:

1. **Main request queue** — `getRequests` in `app/(dj)/events/[code]/page.tsx`.
2. **Pre-event pending-review list** — `getPendingReview` in
   `app/(dj)/events/[code]/components/PreEventVotingTab.tsx`.

Converting only the main list would leave its sibling tab waiting ~5s on a server
round-trip for every sort — the exact inconsistency this issue exists to remove.
**Both lists convert** and share the same sort/filter helpers so they order
identically.

### Out of scope (deliberately)

- `(dj)/dashboard` events list (`getEvents`) and `(dj)/setbuilder` sets list
  (`listSets`) — already load whole arrays; no server pagination to reverse and no
  new sort/filter UX is being added here.
- `admin/users`, `admin/events` (`PaginatedResponse<…>`) — genuinely **unbounded**
  cross-org lists; stay server-paginated (per #489's own carve-out).
- Kiosk display pagination (#478 PR4) — public unattended projector; stays
  server-side growing-window.
- All backend / SSE-payload / migration work — the coalesced-refetch live-update
  choice keeps this PR frontend-only.

## Architecture

Extract the request-list data concern into **shared pure helpers** plus **thin
per-list orchestration**. We touch only the request-store slice of `page.tsx`;
the other ~60 `useState`s (display settings, bridge, Tidal, etc.) are left alone —
splitting those is #506's job and editing them here would collide with a future
#506 PR.

### Shared, reused pure units

| File | Purpose | Depends on |
|---|---|---|
| `dashboard/lib/camelot.ts` | Pure port of server `parse_key` → Camelot ordinal (`number*2 + (B?1:0)`); null/unparseable → sorts last | nothing |
| `dashboard/lib/request-sort.ts` | `sortRequests(rows, field, dir)` for the 7 simple fields; `computeStatusCounts(rows)`; `filterByStatus(rows, status)` | `camelot.ts` |
| `dashboard/lib/load-all-pages.ts` | Chunked loader: pages a `{requests, total}` fetcher 500-at-a-time to the 2000 cap; returns `{requests, total, capped}` | nothing |

`request-sort.ts` is typed over a **minimal structural row interface** —
`{ id, created_at, accepted_at, vote_count, bpm, musical_key, song_title, artist, status }`
— so the same comparators serve both `SongRequest` (main) and `PendingReviewRow`
(pending tab). No flags, no row-type branching.

### Per-list thin orchestration

| File | Role |
|---|---|
| `dashboard/lib/use-event-requests.ts` | Main-queue hook: `loadAllPages(getRequests)` + coalesced SSE refetch + client sort/filter/counts |
| `dashboard/app/(dj)/events/[code]/page.tsx` | Swap server-window state for the hook |
| `dashboard/app/(dj)/events/[code]/components/PreEventVotingTab.tsx` | Refactor to compose the shared helpers; drop "Load More" for load-all-to-cap |

DRY boundary: the **decisions** (Camelot ordering, per-field comparators + exact
tie-breaks, chunked load, 2000 cap) are shared; the **orchestration** (which
fetcher, which row shape, which server-authoritative sort, SSE wiring) stays
per-list. This avoids the "one parameterized hook juggling two row shapes and two
sort enums" knot.

## Sort parity (correctness-critical)

The JS comparators must mirror `server/app/services/request_sort.py` **exactly**,
or rows "jump" between the optimistic-patch view and the post-refetch view (reads
as flicker/bugs):

- Default direction per field: `date_requested` DESC, `date_accepted` DESC,
  `upvotes` DESC, `bpm` ASC, `key` ASC, `title` ASC, `artist` ASC.
- Every sort ends with a deterministic **`id DESC`** tie-breaker.
- Nullable fields (`accepted_at`, `bpm`) → **nulls last in both directions**.
- `title`/`artist` compare **case-insensitively** (`toLowerCase`).
- `key` sort tuple mirrors `key_sorted`:
  `(isNull, signedOrdinal, title.toLowerCase, artist.toLowerCase, -id)`.

## Server-authoritative sorts (the one per-list difference)

Two sorts need server context and are **never client re-sorted** — selecting one
fetches the full set in server order and renders as-is:

- **Main queue:** `best_match` (now-playing harmonic/BPM + priority scorer).
- **Pending tab:** `review_order` (server default; `toPendingReviewParams` returns
  `{}` for it).

All other (7 simple) fields are client-sorted from the in-memory set.

## Data flow

1. **Initial load** — `loadAllPages(fetcher, { status })` pages in 500-row chunks
   (`offset 0/500/1000/1500…`, `limit = PUBLIC_PAGE_MAX = 500`) until `total` is
   reached or the 2000 cap. Returns `{ requests, total, capped }`.
2. **Sort** (7 simple fields incl. Camelot `key`) → in-memory `sortRequests`,
   **zero network**. (`best_match`/`review_order` → server-ordered fetch.)
3. **Status filter + tab counts** → from the in-memory set → correct by
   construction when `total ≤ 2000`.
4. **Live updates** — any SSE event (`request_created`,
   `request_status_changed`, `requests_bulk_update`, metadata) → schedule a
   **coalesced refetch** (~500 ms debounce; a burst collapses to one
   `loadAllPages`). Existing optimistic local patches (vote/status `setRequests`)
   stay for snappiness; the refetch is the source of truth. An `AbortController`
   cancels in-flight chunk loads when a newer refetch / status / sort starts.

## Safety cap

2000-row ceiling. If `total > 2000`: load the first 2000 (server default order),
set `capped`, render a banner — *"Showing 2000 of N requests — sort/filter limited
to these."* The envelope's `total` stays honest. Same cap + banner on both lists.

## Error handling

- Chunk-load failure → keep last-good store; surface via the existing
  `actionError` channel (main) / the tab's error state (pending).
- SSE reconnect is already handled by `useEventStream` (exponential backoff).

## Testing (vitest)

- `camelot.ts`: ordinal mapping, sharps/flats/enharmonics, null/unparseable → last.
- `request-sort.ts`: each of 7 fields asc/desc with server tie-break parity;
  `computeStatusCounts`; `filterByStatus`.
- `load-all-pages.ts`: stitches multiple 500-row pages; stops at `total`;
  `total > 2000` → `capped = true` and exactly 2000 rows; single-page short event.
- `use-event-requests.ts` (fake timers): N SSE events in window → **1** refetch;
  `best_match` → no client re-sort + server fetch; status/sort change aborts an
  in-flight load.
- `PreEventVotingTab.test.tsx`: update fixtures; assert client sort is instant
  (no fetch), `review_order` → server-ordered, "Load More" removed, `capped`
  banner.

## Acceptance criteria

- [ ] Both event-page lists sort/filter instantly client-side with no per-toggle
      fetch (except `best_match` / `review_order`).
- [ ] Tab counts/totals come from the in-memory set (exact when `total ≤ 2000`).
- [ ] Full set loads via 500-row chunks up to 2000; `> 2000` shows the banner.
- [ ] Client sort output matches `request_sort.py` ordering (incl. tie-breaks).
- [ ] Live updates land via coalesced refetch; no per-event refetch storm.
- [ ] No backend/SSE/migration changes; kiosk + admin lists untouched.
- [ ] `npm run lint`, `npx tsc --noEmit`, `npm test -- --run` green; vitest
      coverage gate (68% br / 65% fn / 78% ln / 77% st) maintained.
